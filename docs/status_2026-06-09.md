# DeepThinkTrader — Status 2026-06-09

## TL;DR

A full code and logic review (six parallel review passes over ~16k lines, every major
claim re-verified against source and data) found that the quality_momentum edge never
existed: a timezone bug in the backtest suite backdated 99% of simulated entries by a
median ~2-4 hours, inflating the signal to PF 1.5. With the bug fixed, quality_momentum
is PF 0.98 — dead-center no-edge — and a corrected re-run of the full 8-signal search
shows **nothing with a demonstrable edge on either book**. The penny book is decisively
negative once entries are simulated honestly. Main is halted (`MAIN_ENABLED=false`,
bot restarted clean). The bot is now a healthy platform with no strategy, by design.
The next move is strategic: a genuinely different signal class, tested on the
now-trustworthy validator, or repurposing the infrastructure.

## The timezone bug (the finding that matters)

`research_reports.timestamp` (and `trades.timestamp`) are written with
`datetime.now().isoformat()` — naive **local Eastern**. Backtest bars are naive **UTC**
(`trade_replay.py` converts yfinance timestamps explicitly). `_parse_ts` passed naive
input through unconverted, so the validator's "next bar strictly after the report"
comparison treated 2:00 PM ET as 2:00 PM UTC and entered at the 10:30 AM bar — hours
before the signal existed, on stocks selected because they were up on the day.

Verified empirically before fixing: 395 of 398 simulated entries were backdated.
Cache-only re-run of the exact pipeline, one change (report timestamps ET→UTC):

| Run | n | win % | expectancy | PF |
|---|---|---|---|---|
| As shipped (skewed) | 312 | 58.0 | +0.72% | 1.51 |
| Timezone-corrected | 278 | 52.5 | −0.02% | 0.99 |

Blast radius: every backtest module parses timestamps through `_parse_ts` —
`signal_search` (the 16-combo search that crowned quality_momentum),
`walk_forward_entries` (the conviction-stack halt numbers; that halt remains
directionally right since the live −$8.3k ledger independently corroborates it),
`penny_regime`, `candidate_config` (exit-policy and gate selections).

Two compounding problems found alongside it: the validator's "walk-forward OOS"
computes in-sample aggregates but never selects anything, so its "OOS PF 1.35" was
full-history-minus-28-days relabeled; and no significance testing existed anywhere —
at this variance, n≈200 cannot distinguish PF 1.35 from 1.0 even on clean data.

**Fix shipped:** `_parse_ts` now localizes naive timestamps America/New_York → UTC
(commit `200205f`, pushed), with a regression test covering EDT, EST, and
already-aware inputs (`tests/test_trade_replay.py`). All 24 backtest tests pass.

## Also established by the review

- **The "live forward test" never ran.** All 1,170 BUY signals since 6/04 were blocked,
  ~85% by the drawdown halt — the 30-day peak ($104,583) was set by the old strategy in
  late May (`BASELINE_DATE` defaults to 2026-03-24), so the new strategy inherited an
  11.4% drawdown it could never trade out of. Only 2 main trades ever closed. The halt
  accidentally prevented trading a no-edge strategy; it stays as-is.
- **The deployed rule wasn't the backtested rule anyway:** `edges_ok` is re-applied
  downstream in `risk_manager.validate_trade` (the bypass there is the manual-order
  flag, not signal mode), live SMA-20/up-day use intraday partial data where the
  backtest used completed bars, and the backtest exit model omits scale-outs, the time
  stop, earnings auto-close, momentum stop-tighten, and gap-through fills.
- **Money-path bugs to fix before trusting any future run:** `trades.pnl` excludes
  partial-exit P&L (corrupts Kelly sizing, edge stats, auto-pause, revenge detection);
  penny limit orders are recorded as filled with no fill polling (unfilled orders later
  fabricate phantom P&L via reconcile); no `client_order_id` idempotency and reconcile
  is one-directional (a timed-out entry can double-buy and leave an unmanaged Alpaca
  position with no stop, ever); exits are recorded at the pre-close quote, not the fill.
- **Safety gaps:** paper-only is a startup *warning*, not an error — one env var plus
  live keys away from live trading; all Slack notifications are no-ops
  (`NOTIFICATIONS_ENABLED` unset), so halts and 5 days of blocked trades were invisible;
  the unguarded main loop has crash-respawned silently hundreds of times (362 tracebacks
  in `bot_stderr.log`, mostly sqlite CANTOPEN from per-30s `Database()` churn — the
  "mystery restarts" explained).
- **Data-quality issues:** live Alpaca bars are fetched `adjustment:"raw"` (reverse-split
  penny stocks read as 10x winners in relative strength); the report cache can serve a
  stale up-day flag for up to 4 hours; the yfinance "timeouts" don't actually time out
  (`with ThreadPoolExecutor` joins the hung thread); `market_clock` serves a stale
  `is_open` forever if the Alpaca clock API fails; `tunable_params.json` silently
  overrides `.env` for 11 risk params.
- **Sound and verified:** per-book routing inside `analyze()`, the 6/06 regime-thresholds
  fix, exit comparison directions both sides, tunable bounds + audit logging, per-user
  error isolation, secrets encryption, read-only discipline in the backtest suite.

## What would have to happen before any strategy trades again

1. A signal from a genuinely different class (post-earnings drift, insider activity,
   event-driven — something not derivable from the mover-biased scanner universe), since
   the current 15.5k-report dataset has now been mined twice and anything "found" in it
   needs fresh forward data to count as evidence.
2. It passes the corrected validator: next-bar entry, realistic cost, bootstrap CI
   excluding 1.0, consistency across periods.
3. The money-path bugs above are fixed first, so live results are measurable.
4. Several hundred independent trades of live paper evidence — not 30-50; at observed
   per-trade variance, distinguishing PF 1.2 from 1.0 needs n in the hundreds.

## Honest caveats

- "No edge in this data" is not "no edge exists" — 2.5 months of one regime, on a
  scanner-conditioned universe, can't rule out signals that work elsewhere. But the
  burden of proof now sits where it should: on future signals, against a validator
  that no longer lies.
- The corrected validator still simplifies exits (no scale-outs/time-stop/earnings
  exits) and assumes fills at the next bar's open. Good enough to kill bad ideas;
  a survivor would still need the live exit engine modeled before sizing decisions.

## Evening addendum — money-path fixes shipped, 8 orphan positions found and closed

### Money-path fixes (commit `3a23c37`, pushed; bot restarted 21:24)

All four money-path bugs from the morning review are fixed, with red-green tests
(12 new in `tests/test_money_path_fixes.py`; suite 226 pass, same 7 pre-existing
failures):

1. **`trades.pnl` includes partial-exit P&L.** `close_trade` sums `partial_exits`
   into the stored pnl, so a scaled-out winner records as a winner. Kelly stats,
   `edge_performance`, the auto-pause, and revenge detection now read true
   outcomes. Daily P&L doesn't double-count — callers still pass the final leg
   only, since partials were booked on their own day.
2. **Limit entries settle to actual fills before recording.** Poll 60s → cancel
   remainder → one re-read for fills that race the cancel. Zero fill = no trade
   row (the ghost-row → fabricated-P&L reconcile chain is dead). The
   `_poll_order_status` timeout now reports observed fills instead of zero.
3. **`client_order_id` on every order POST** (market, limit, retry, scale-out)
   with transport-failure recovery via `/v2/orders:by_client_order_id` — a
   timed-out-but-accepted order can no longer create an untracked position or a
   double-buy. `reconcile_open_trades` also checks the reverse direction now and
   logs ERROR for Alpaca positions with no DB row (never auto-touched — they
   could be deliberate manual positions).
4. **Exits record realized fills, not pre-close quotes.** Full exits,
   `manual_close`, and scale-outs resolve the liquidation/child order's
   `filled_avg_price` (snapshot fallback with a logged warning). The
   `alpaca_reconcile_retry` path now writes `daily_pnl` (it never did).
